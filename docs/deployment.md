# Deployment

## What Deployment Means Here

There is no single release pipeline. Deployment splits into three tracks:

| Track | What changes | How it deploys |
|---|---|---|
| Container image | `novnc-desktop/` source | git push → GitHub Actions → GHCR → manual `docker pull` |
| Host scripts and configs | keeper scripts, oauth2-proxy, traefik | git push → `git pull` on server → manual service restart |
| Runtime config | `workspace/openclaw.json` | direct file edit or keeper script convergence |

hiclaw-manager and hiclaw-controller are **not in Coolify** and are **not rebuilt from this repo**. They come from Alibaba's registry and are kept alive by host-side keeper scripts running as cron jobs.

## Persistent Boundary

Everything on the host filesystem at `/worksp/hiclaw/` survives container recreation. Everything inside containers that is not backed by a bind mount or a keeper script does not.

| Survives container recreation | Does not survive |
|---|---|
| `workspace/` content (bind-mounted to `/root/manager-workspace` in manager) | `docker exec`-made changes with no keeper backing |
| `workspace/openclaw.json` | npm packages installed inside container overlay |
| Host keeper and startup scripts | `/usr/local/bin/mc` wrapper (installed by `fix-element-config.sh`; re-installed on next controller restart) |
| `oauth2-proxy/` config | `/usr/local/bin/npm` wrapper (reinstalled by `start-manager-agent.sh` on startup) |
| `traefik/claw.yml` | nginx configs generated inside controller (regenerated by `start-element-web.sh` on restart) |
| `novnc-desktop/` volume (`novnc-e10kwzww46ljhrgz1qj08j6a-config`) | clawtalk in-container modifications (see AGENTS.md § Pending Work) |

## Required Host Cron Jobs

All four must be present in the `ai` user's crontab. Verify with `sudo crontab -u ai -l`.

```cron
* * * * * /worksp/hiclaw/manager-config-keeper.sh >> /worksp/hiclaw/manager-config-keeper.log 2>&1
* * * * * /worksp/hiclaw/manager-bootstrap-keeper.sh >> /worksp/hiclaw/manager-bootstrap-keeper.log 2>&1
* * * * * /worksp/hiclaw/controller-bootstrap-keeper.sh >> /worksp/hiclaw/controller-bootstrap-keeper.log 2>&1
* * * * * /worksp/hiclaw/novnc-resource-keeper.sh >> /worksp/hiclaw/novnc-resource-keeper.log 2>&1
```

If any are missing, add them with `sudo crontab -u ai -e`. Missing keepers mean:
- manager startup script edits never reach the running container
- memory limits are never re-applied after container recreation
- novnc-desktop can run uncapped and cause host-wide Chrome/QtWebEngine OOM
- openclaw update marker is never consumed (UI "Update now" does nothing)
- openclaw.json drifts (wildcard groups key, stale contextWindow, observe-recovery reverts)

## Container Resource Limits

| Container | RAM | Total swap (--memory-swap) | CPUs | PID limit |
|---|---|---|---|---|
| `hiclaw-manager` | 1536 MiB | 3072 MiB (~1.5 GiB of actual swap) | 1 | Docker default |
| `hiclaw-controller` | 3072 MiB | 4096 MiB (~1 GiB of actual swap) | 2 | 1024 |
| `novnc-desktop` | 3072 MiB | 4096 MiB (~1 GiB of actual swap) | 2 | 250 |

**Why manager swap is set high:** Docker's `--memory-swap` is the *total* of RAM + swap — not additive. With `--memory 1536m --memory-swap 1536m`, actual swap is 0 bytes. npm install during an openclaw update peaks well above 1536 MiB total and OOM-kills mid-install, leaving a broken partial install. Setting `--memory-swap 3g` gives ~1.5 GiB of actual swap headroom.

## Host Memory Policy

The host swapfile is `/swapfile`, mounted from `/etc/fstab`, sized at 12 GiB. Swappiness is pinned to `20` in `/etc/sysctl.d/99-hiclaw-memory.conf`.

This is a host-level safety valve only. The first line of defense is still container limits, especially `novnc-desktop` (`3g` RAM, `4g` total RAM+swap, 250 PIDs). Do not remove those limits just because the host has more swap; without them Chrome/QtWebEngine can still drive a global OOM and make SSH unreachable.

The keeper re-applies these limits after every container recreation:

```bash
# manager
docker update --memory 1536m --memory-swap 3g --cpus 1 hiclaw-manager

# controller
docker update --memory 3g --memory-swap 4g --cpus 2 --pids-limit 1024 hiclaw-controller

# novnc desktop
docker update --memory 3g --memory-swap 4g --cpus 2 --pids-limit 250 novnc-desktop
```

## Starting the System

### First-time or after both containers are stopped

```bash
cd /worksp/hiclaw

# Start controller first — MinIO, Matrix, and Element Web must be up before manager needs them
docker start hiclaw-controller
sleep 20

# Check for MinIO recursion BEFORE starting the manager (see architecture.md § MinIO Sync Safety)
sudo find /var/lib/docker/volumes/hiclaw-data/_data/minio/hiclaw-storage \
  -maxdepth 8 -type d -name "hiclaw-storage" -print
# Must print only the root path. Extra lines = recursion bug present.
# If extra lines appear: do NOT start manager — follow architecture.md § MinIO recovery.

docker start hiclaw-manager
sleep 30

# Verify MinIO still clean after manager startup
ls /worksp/hiclaw/workspace/hiclaw/ 2>/dev/null && echo "WARNING: recursion seed appeared" || echo "OK"

# Verify manager
docker exec hiclaw-manager openclaw clawtalk doctor
bash /worksp/hiclaw/manager-bootstrap-keeper.sh
```

### novnc-desktop (or full recreate)

```bash
# Shortcut: bash /worksp/hiclaw/novnc-desktop/recreate.sh

docker pull ghcr.io/u2giants/novnc-desktop:latest
docker stop novnc-desktop && docker rm novnc-desktop
docker run -d --name novnc-desktop \
  --network e10kwzww46ljhrgz1qj08j6a --ip 10.0.5.4 \
  --dns 1.1.1.1 --dns 8.8.8.8 \
  --memory 3g --memory-swap 4g --cpus 2 --pids-limit 250 \
  -v novnc-e10kwzww46ljhrgz1qj08j6a-config:/config \
  -e PUID=1000 -e PGID=1000 -e TZ=UTC -e "TITLE=HiClaw Desktop" \
  --shm-size=2g --restart unless-stopped \
  ghcr.io/u2giants/novnc-desktop:latest
docker network connect coolify novnc-desktop
```

`--dns 1.1.1.1 --dns 8.8.8.8` is required: the host resolver uses Tailscale (`100.100.100.100`) which is unreachable from inside Docker containers — all DNS lookups fail without explicit public resolvers.

`--ip 10.0.5.4` is required: this address is hardcoded as the CDP endpoint for Playwright MCP.

**Rollback:** replace `:latest` with `:sha-<previous-commit>` in the `docker run` command.

### oauth2-proxy

```bash
cd /worksp/hiclaw/oauth2-proxy && docker compose up -d
```

## Normal Container Recreation (automatic)

When HiClaw recreates `hiclaw-manager` or `hiclaw-controller`, the keeper cron jobs converge within 60 seconds:

1. HiClaw creates a new container with the stock startup script.
2. The keeper detects the startup script hash mismatch (or new container ID).
3. The keeper `docker cp`s the host-owned patched script into the container and re-applies resource limits. No immediate restart — the patch takes effect on the container's next natural restart.
4. If the openclaw package hash also changed since the last boot, the keeper triggers `docker restart hiclaw-manager`.
5. On restart, `start-manager-agent.sh` runs: MinIO sync, ClawTalk bootstrap, whatsapp bootstrap, openclaw hash write, `exec openclaw gateway run`.

The keeper logs `container startup script not readable yet; skipping this run` while the container is still booting. This is expected.

### What the keeper does and does not restart for

| Change | Restart |
|---|---|
| `start-manager-agent.sh` content changed | No — patched silently; takes effect on next natural restart |
| openclaw `package.json` hash changed (after update) | Yes — `docker restart hiclaw-manager` |
| Container recreated with same scripts | Resource limits re-applied; no extra restart |
| `start-element-web.sh` or `start-tuwunel.sh` hash changed | Yes — `docker restart hiclaw-controller` |

A `git push` of a script change does not interrupt the active user session.

## openclaw Updates

### Normal flow (via Control UI)

1. Open `https://gateway.claw.designflow.app`
2. Click "Update now"
3. openclaw calls the fake `/usr/local/bin/systemd-run` (installed by `start-manager-agent.sh`), which writes `/worksp/hiclaw/workspace/.openclaw-update-requested` and exits 0
4. openclaw does an in-process SIGUSR1 restart — the container stays running
5. Within 60 seconds, `manager-bootstrap-keeper.sh` detects the marker, removes it, and runs inside the container:
   ```bash
   docker exec hiclaw-manager openclaw update --yes --json
   ```
   with the container already set to `--memory 1536m --memory-swap 3g`
6. The keeper **waits 30 seconds** after the update command completes, allowing the in-process SIGUSR1 restart (including openclaw.json write) to finish before proceeding
7. The keeper then compares the current in-container `package.json` hash against the hash written at startup; if changed, it runs `docker restart hiclaw-manager`
8. On restart, `start-manager-agent.sh` validates the npm install (checks `json5/package.json` and `openai/index.mjs`), updates `/usr/local/bin/openclaw` to point to the npm binary, records the new hash

Verify after restart:

```bash
docker exec hiclaw-manager openclaw --version
```

### Why the 30-second wait matters

openclaw uses an in-process SIGUSR1 restart (not exec-based, due to `OPENCLAW_NO_RESPAWN=1`). This restart includes a write phase where openclaw rewrites `openclaw.json` with the updated config. If `docker restart` is issued during this write window, the container is killed mid-write and leaves a truncated config file on the bind-mounted volume — a broken state that crashes the gateway on every subsequent startup.

The 30-second sleep in the keeper absorbs this window. **Do not shorten or remove it.**

### openclaw install validation

The startup script checks two files before symlinking the npm-installed openclaw:
- `/usr/lib/node_modules/openclaw/node_modules/json5/package.json`
- `/usr/lib/node_modules/openclaw/node_modules/openai/index.mjs`

If either is absent (OOM-interrupted partial install), the broken directory is deleted and the symlink falls back to the base image at `/opt/openclaw/`. This prevents the gateway from crashing at startup with `Cannot find module openai/index.mjs`.

### openclaw hash detection

`start-manager-agent.sh` writes a sha256 of the active openclaw `package.json` to:

```
/worksp/hiclaw/workspace/.openclaw-startup-pkg-hash   (host path)
/root/manager-workspace/.openclaw-startup-pkg-hash    (container path — same file via bind mount)
```

`manager-bootstrap-keeper.sh` reads the host copy and compares it against the current in-container package hash. Both sides probe `/usr/lib/node_modules/openclaw/package.json` first, then fall back to `/opt/openclaw/package.json`.

### Manual update (emergency only)

Only do this if the normal marker flow is not available.

```bash
# 1. Pause keeper crons
sudo crontab -u ai -e    # comment out keeper lines

# 2. Ensure swap headroom
docker update --memory 1536m --memory-swap 3g --cpus 1 hiclaw-manager

# 3. Run update
docker exec hiclaw-manager openclaw update --yes --json

# 4. Wait for in-process SIGUSR1 to finish
sleep 30

# 5. Re-enable crons
sudo crontab -u ai -e    # uncomment keeper lines

# 6. Trigger keeper to detect hash change
bash /worksp/hiclaw/manager-bootstrap-keeper.sh
```

## Environment Variables

**Manager env vars** are passed at `docker run` time from `start-manager-agent.sh`. The critical secrets (`HICLAW_MANAGER_GATEWAY_KEY`, `HICLAW_MANAGER_PASSWORD`) are auto-generated on first run and persisted to `/data/hiclaw-secrets.env` (chmod 600, bind-mounted from the host).

**Controller env vars** are set in the `docker run` command inside `start-manager-agent.sh` for the controller, and in the Coolify service definition for services that are Coolify-managed.

**novnc-desktop env vars** are set by the `docker run` command (see "Starting the System" above) and stored in the named volume.

**All root variable names** are documented in `.env.example`; oauth2-proxy variables are documented in `oauth2-proxy/.env.example`. Root `.env` and `oauth2-proxy/.env` are ignored.

| Secret category | Storage location |
|---|---|
| Admin password (Higress + MinIO) | `/data/hiclaw-secrets.env` on host |
| OpenRouter API key | `/data/hiclaw-secrets.env` |
| Manager auth (gateway key, password, JWT) | `/data/hiclaw-secrets.env` |
| Matrix registration token | `/data/hiclaw-secrets.env` |
| MinIO secret key | `/data/hiclaw-secrets.env` |
| Google OAuth credentials | `oauth2-proxy/.env` (server-local, ignored by git) |
| openclaw API key + Matrix token | `workspace/openclaw.json` (runtime, not committed) |

## nginx Upstream for control.claw

nginx inside `hiclaw-controller` proxies `control.claw.designflow.app` → `hiclaw-manager:18799`. It must not use a hardcoded IP — Docker reassigns IPs on container recreation.

`start-element-web.sh` generates `manager-console.conf` with Docker's embedded DNS resolver:

```nginx
resolver 127.0.0.11 valid=10s;
set $upstream hiclaw-manager;
proxy_pass http://$upstream:18799;
```

By storing the hostname in a variable, nginx bypasses its startup-time DNS cache and re-resolves on each request.

If `control.claw.designflow.app` returns 502 after a manager restart:

```bash
# Verify the config is present and uses the variable form
docker exec hiclaw-controller cat /etc/nginx/conf.d/manager-console.conf

# If missing or hardcoded: restart the controller to regenerate it
docker restart hiclaw-controller
```

## Model Context Window Sync

On each manager startup, `start-manager-agent.sh` fetches the live model list from OpenRouter and updates `contextWindow` and `maxTokens` for matching model IDs in `openclaw.json`. The sync writes back to MinIO immediately so a subsequent MinIO→local pull does not overwrite the fresh values.

Models accessed via Higress gateway alias IDs (e.g. `gpt-5.4`, `gpt-5-mini`) do not match OpenRouter IDs and keep their statically configured values. The deepseek models enforce a minimum context window via `manager-config-keeper.sh` even if the reconciler reverts them.

## Manual Recovery

If automation has not converged after two minutes:

```bash
bash /worksp/hiclaw/manager-bootstrap-keeper.sh
bash /worksp/hiclaw/manager-config-keeper.sh
bash /worksp/hiclaw/mcp-keeper.sh
docker exec hiclaw-manager openclaw clawtalk doctor
```

If `openclaw.json` is corrupt or truncated (symptoms: gateway crashes at startup, keeper logs parse errors):

```bash
# Option 1: restore from last-good backup
cp /worksp/hiclaw/workspace/openclaw.json.bak /worksp/hiclaw/workspace/openclaw.json
docker restart hiclaw-manager
```

If the `.bak` file is also bad — check file size first (healthy `openclaw.json` is ~8.9 KB or larger; a truncated file is a few hundred bytes):

```bash
ls -lh /worksp/hiclaw/workspace/openclaw.json /worksp/hiclaw/workspace/openclaw.json.bak

# Option 2: restore from MinIO (reflects last successful startup sync)
docker exec hiclaw-manager mc cp hiclaw/hiclaw-storage/manager/openclaw.json \
  /root/manager-workspace/openclaw.json
docker restart hiclaw-manager
```

## Do Not Do

These actions will break the system:

| Action | Why it breaks things |
|---|---|
| `docker exec hiclaw-manager openclaw update --yes` without pausing keepers | Config keeper fires mid-install and modifies openclaw.json; npm OOM risk without swap check |
| `docker restart hiclaw-manager` immediately after an openclaw update (within 30s) | Kills the container mid-write during the in-process SIGUSR1 restart; produces a truncated openclaw.json that crashes the gateway on every subsequent startup |
| `docker update --memory-swap 1536m hiclaw-manager` (same value as `--memory`) | Sets actual swap to 0 bytes; npm install OOM-kills the container during openclaw updates |
| Write or `cp` to `novnc-desktop/cdp_proxy.py` on the host | Creates a new inode; Docker bind mount still points to old inode; container never sees the update. Use Edit tool or `sed -i` instead. |
| Set `channels.matrix.groups["*"]` in openclaw.json | Rejected by OpenClaw schema; breaks `update.run` and every config reload |
| Change `commands` away from `{"restart": true}` in openclaw.json | Gateway triggers restart on next reconciler diff; causes a restart loop |
| Deploy hiclaw-manager or hiclaw-controller via Coolify | They are not in Coolify. Coolify will not find them and will create duplicates. |
| Run `openclaw update` directly inside the container while the marker mechanism is working | Bypasses swap-headroom enforcement and the 30s grace window; risks truncation and conflicts with keeper |
| Set `--memory 1536m` without also increasing `--memory-swap` above 1536m | Equivalent to setting swap to 0 bytes; same OOM failure as above |

## Release / Change Management

1. Edit on the host or in a local branch
2. Validate against the running system
3. Commit with a clear imperative message (see CLAUDE.md § Commit Style)
4. Push to `main`
5. Pull on server: `cd /worksp/hiclaw && git pull`
6. Apply (see Script and Config Changes section above)

No separate publish step for shell scripts or configs. Only `novnc-desktop/` changes trigger a CI build.
